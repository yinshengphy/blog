Product & Architecture Report

Blog 智能机器人设计方案

为个人 Astro 博客增加一个右下角极简聊天机器人。它能够识别用户意图，在人格问答、博客 RAG 问答和通用问答之间分流；当命中博客内容时展示来源，并允许用户跳转到对应文章。

部署约束
K3s 单节点，现有博客仍保持静态 Astro + nginx 服务。
核心服务
blog-service，Spring AI 后端，独立 Deployment。
本地模型
llama.cpp server + RTX 2060 6GB 量化模型。
RAG 存储
Qdrant 存储已发布博客正文切片与来源元数据。

1. 设计目标

降低阅读摩擦

访客无需翻完整站，就能直接询问博客中讲过的知识点，并从答案跳回文章。

保留个人感

人格问答由独立人设资料驱动，用来回答“你是谁”“这个站为什么存在”等问题。

本地优先

模型和向量库自托管，避免云模型成本和外部依赖，符合当前 K3s 私有部署方式。

已确认的产品边界

问题 决策 设计影响
模型部署 本地 llama.cpp server 需要独立模型服务、GPU 资源限制、冷启动和并发保护。
RAG 范围 只检索已发布博客正文 索引管线只读取 Astro content posts，来源可以稳定映射到文章 URL。
人设资料 后续单独提供 作为独立 persona.md 或配置项进入后端，不混入博客 RAG。
开放策略 访客可问，基础限流 需要 IP hash + SQLite 时间窗口限流，不保存完整问题。
交互形式 极简右下角聊天窗 不做完整客服系统，降低页面干扰。
来源展示 只在命中博客内容时展示 人格和通用回答不展示来源，RAG 回答底部展示来源卡片。

2. 页面与交互设计

机器人入口采用右下角浮动按钮。展开后是小聊天窗，不覆盖整页阅读；移动端宽度接近全屏但仍保留边距。 视觉风格沿用博客当前的简约浅色系统：低饱和背景、轻边框、8-14px 圆角、少量阴影。

博客助手 可以问文章内容、作者信息或通用问题

这属于博客内容问题。我会先检索相关文章，再结合命中的段落回答...

AI

推荐交互流程

  1. 初始态

    右下角显示小型 AI 按钮。首次访问可用一个短 tooltip 提示“问问博客内容”。

  2. 展开聊天窗

    顶部显示“博客助手”和能力说明；正文区展示 2-3 个建议问题；底部为输入框。

  3. 发送问题

    前端通过 SSE 调用 /api/assistant/chat，答案流式输出。

  4. 展示来源

    只有 RAG 命中博客内容时，答案下方出现来源卡片，点击跳转到文章 URL 或章节锚点。

  5. 错误与限流

    模型忙、限流、检索为空时使用短文案解释，不暴露内部错误。

关键状态

空状态

展示能力边界和建议问题，例如“这个博客讲过哪些密码学内容？”“作者是谁？”

生成中

SSE 逐段输出，输入框禁用或允许停止生成。建议保留“停止”按钮。

RAG 无命中

不强行编造来源。提示“博客里没有找到可靠内容，我按通用知识回答”。

模型繁忙

RTX 2060 资源有限，排队或返回“当前繁忙，请稍后再试”。

3. 技术架构总览

现有博客继续保持静态部署。新增后端命名为 blog-service，原因是它未来不仅服务智能机器人， 也可以承载评论、订阅、搜索、统计等博客后端能力。为了避免后续混乱，当前实现中把智能机器人作为 assistant 模块或 bounded context。 

用户浏览器
  ├─ GET /                    → blog-web nginx → Astro 静态文件
  └─ GET /api/assistant/chat  → blog-service Spring Boot
                                ├─ Intent Router
                                ├─ Persona Answer Chain
                                ├─ Blog RAG Chain
                                │   ├─ Qdrant: blog_chunks collection
                                │   └─ Source Builder: article URL + heading
                                ├─ General Answer Chain
                                ├─ SQLite: rate_limit + metrics
                                └─ llama.cpp server: local GGUF model

推荐服务命名

资源 推荐名称 说明
Spring Boot 后端 blog-service 面向博客后端能力的统一服务，避免未来新增评论/订阅时名字过窄。
AI 模块包名 assistant 内部模块边界，包含意图识别、RAG、人格问答和通用问答。
模型服务 llama-server 部署 llama.cpp server，暴露 OpenAI-compatible API。
向量库 qdrant 独立 StatefulSet 或 Deployment + PVC。
索引任务 blog-indexer CI 或手动 Job，用于生成博客切片和 embeddings。

4. 技术选型

层级 选型 理由 风险
后端 Spring Boot + Spring AI 符合你的偏好；Java 生态稳定；Spring AI 提供 Chat、Embedding、VectorStore、RAG Advisor 抽象。 Spring AI 版本迭代较快，落地前需锁版本并验证 API。
模型服务 llama.cpp server 轻量、可控、适合 GGUF 量化模型；适合 RTX 2060 6GB 和低资源环境。 并发有限；模型效果依赖选型和量化质量。
向量库 Qdrant 独立服务、扩展性好、支持 payload 元数据和过滤，适合来源追踪。 比文件型向量库多一个服务，需要持久化和备份。
索引 CI 自动索引 + 手动 reindex 发布文章后索引同步更新；手动入口方便修复和重建。 CI 时间增加；模型服务不可用时索引可能失败。
响应 SSE 实现简单，适合文本流式输出；比 WebSocket 更贴合单请求生成。 需要处理断开、超时和前端停止生成。
限流 SQLite 轻量、持久化、够个人博客使用；记录 IP hash 时间窗口即可。 多副本时需共享 PVC 或改为 Redis；初期建议 blog-service 单副本。
重要约束： 你的现有博客是静态站，不能把模型 Key、RAG 检索或限流逻辑放在 Astro 前端。前端只能是聊天组件， 所有智能逻辑都应进入 blog-service。

5. RAG 与意图识别设计

意图类型

类型 示例 处理链路 是否展示来源
persona 你是谁？这个站是谁写的？作者现在关注什么？ 读取人设资料 + 受控回答模板 否
blog_rag RSA 为什么需要欧拉函数？这篇文章讲了什么？ Query embedding → Qdrant 检索 → 上下文压缩 → 生成回答 是
general 解释一下 HTTPS；推荐一个学习路线。 直接调用本地模型，使用通用系统提示词 否
reject_or_clarify 请求隐私、攻击、无意义输入、过长输入。 拒答或追问 否

意图识别策略

初期不建议上复杂分类模型。更稳的方案是“规则优先 + 小模型分类兜底”：

  1. 规则命中人格问题

    包含“你是谁、作者是谁、这个站、yinsheng、联系方式”等关键词时进入 persona。

  2. 规则命中博客问题

    包含文章标题、标签、已知主题词，或用户明确说“博客里/文章中/这篇”时进入 blog_rag。

  3. 轻量 LLM 分类

    无法判断时调用本地模型输出 JSON：{ intent, confidence, reason }。

  4. 低置信度兜底

    低置信度时先做一次 RAG 检索；若得分低于阈值，按 general 回答且不展示来源。

RAG 索引数据结构

切片建议

参数 建议 原因
切片粒度 按 Markdown 标题分段后再按 450-800 中文字符切片 保留章节语义，来源卡片也能定位到标题。
重叠 80-120 中文字符 避免公式或推导链断裂。
召回数量 Top 5 检索，最终给模型 Top 3 降低上下文噪声，适合小模型。
来源展示 最多 3 个来源卡片 移动端空间有限，避免喧宾夺主。

7. K3s 部署方案

新增组件不应破坏现有 blog-web。建议新增以下资源：

资源 类型 建议配置
blog-service Deployment + Service Spring Boot，初期 1 副本，内存 512Mi-1Gi，挂载 SQLite PVC。
llama-server Deployment + Service GPU 调度，挂载模型 PVC，限制并发，暴露集群内 HTTP。
qdrant StatefulSet 或 Deployment + PVC 1 副本，持久化 collection，ClusterIP 仅内网访问。
blog-ai-data PVC 存放 SQLite、必要配置和索引状态。
blog-models PVC 存放 GGUF 模型文件，避免镜像过大。

流量路径

Cloudflare Tunnel
  ├─ /             → blog-web Service:80
  └─ /api/*        → blog-service Service:8080

blog-service
  ├─ http://llama-server:8080/v1/chat/completions
  ├─ http://llama-server:8080/v1/embeddings 或独立 embedding server
  ├─ http://qdrant:6333
  └─ /data/rate-limit.sqlite

模型部署建议

生成模型

RTX 2060 6GB 更适合 3B-7B 量化模型。优先选择 Q4_K_M / Q5_K_M 这类 GGUF 量化。 并发建议从 1 开始，避免显存抖动。

Embedding 模型

最好与生成模型分开。可以使用独立 embedding 模型服务；若显存紧张，embedding 可 CPU 跑， 因为在线只需要对用户 query 做一次 embedding。

注意： 单张 RTX 2060 6GB 同时跑生成模型和 embedding 模型可能紧张。第一版可以让生成模型使用 GPU， embedding 使用 CPU 或离线任务错峰运行。

8. CI/CD 与索引流程

现有 CI 在 npm run build 后复制 dist/ 到 /deploy/public/。 新增智能机器人后，博客发布和索引更新应保持一致。

  1. 构建 Astro

    沿用现有流程：checkout → npm ci → npm run build。

  2. 生成索引输入

    解析 src/content/posts/*.md，过滤 draft，输出 post_id/title/url/heading/text。

  3. 切片与 embedding

    调用集群内 embedding 服务生成向量，或在 CI runner 内执行索引任务。

  4. 写入 Qdrant

    按 collection upsert；使用 source_hash 判断是否需要更新。

  5. 发布静态文件

    复制 dist/ 到 blog-public PVC。

建议的索引命令

# CI 中执行
java -jar blog-service-indexer.jar \
  --posts-dir ./src/content/posts \
  --site-base-url https://你的博客域名 \
  --qdrant-url http://qdrant:6333 \
  --collection blog_posts

9. 安全、隐私与成本控制

限流

SQLite 表记录 ip_hash、分钟窗口、日窗口、失败次数。初期建议 10 次/分钟、100 次/天。

日志

只记录请求量、耗时、意图、是否 RAG 命中、错误码。不保存完整问题正文。

防滥用

限制输入长度，例如 500 中文字符；限制单次输出 tokens；模型繁忙时快速失败。

来源约束

只有 RAG 命中且分数过阈值时展示来源，避免伪来源。

人设保护

人设资料只作为回答边界，不暴露原文；遇到隐私请求应拒答。

公网暴露

只暴露 /api/assistant/chat；Qdrant、llama-server、admin reindex 均不直接暴露公网。

10. 实施路线

阶段 目标 交付物
Phase 1 静态 UI + API 合约 Astro 聊天组件、SSE 前端封装、错误状态、来源卡片样式。
Phase 2 blog-service 基础能力 Spring Boot 服务、SSE chat endpoint、SQLite 限流、基础日志。
Phase 3 本地模型接入 llama-server Deployment、模型 PVC、Spring AI ChatClient 接入。
Phase 4 RAG 上线 Qdrant、索引器、博客切片、来源卡片跳转。
Phase 5 人格与意图分流 persona 资料接入、intent router、RAG/general/persona 三链路。
Phase 6 运维优化 监控、健康检查、重建索引、限流面板或日志报表。

12. 仍需你补充的信息

  • 人设资料：作者身份、回答语气、可公开信息、不能回答的信息。
  • 博客公网域名：用于构造来源 URL。
  • 模型选择：具体 GGUF 模型文件、量化等级、上下文长度。
  • 是否接受生成速度较慢：本地 6GB GPU 对长回答会有限制。
  • Cloudflare Tunnel 当前 ingress 配置：确认是否能 path route 到 blog-service。
