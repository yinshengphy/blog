产品与架构报告

和我聊聊吧~ 设计方案

为个人 Astro 博客增加一个右下角极简问答机器人。访客问一句，它回答一句；回答优先基于博客内容， 并返回可点击来源。页面不展示任何服务器、硬件或内部网络信息，只呈现清晰、克制的问答体验。

部署约束
私有化部署，现有静态博客服务不变。
核心服务
独立 RAG 后端，前端只访问博客自己的 API。
AI 计算
私有模型服务统一调度，具体硬件和内部拓扑不对访客展示。
RAG 存储
保存博客切片、语义向量和可点击来源元数据。

1. 设计目标

降低阅读摩擦

访客无需翻完整站，就能直接询问博客中讲过的知识点，并从答案跳回文章。

保留个人感

人格问答由独立人设资料驱动，用来回答“你是谁”“这个站为什么存在”等问题。

本地优先

模型和向量库自托管，避免云模型成本和外部依赖；访客侧不暴露底层部署细节。

已确认的产品边界

问题 决策 设计影响
模型部署 私有 AI 计算网关 + 私有模型运行时 业务只调用统一网关；页面和公开响应不出现具体硬件型号、显存、内部服务名或网络地址。
RAG 范围 只检索已发布博客正文 索引管线只读取 Astro content posts，来源可以稳定映射到文章 URL。
人设资料 后续单独提供 作为独立 persona.md 或配置项进入后端，不混入博客 RAG。
开放策略 访客可问，基础限流 需要 IP hash + SQLite 时间窗口限流，不保存完整问题。
交互形式 问答式右下角聊天窗 每个用户问题默认只产生一次完整回答；除非内容确实过长，不主动拆成多条。
来源展示 只在命中博客内容时展示 人格和通用回答不展示来源，RAG 回答底部展示来源卡片。

2. 页面与交互设计

机器人入口采用右下角浮动按钮。展开后是小聊天窗，不覆盖整页阅读；移动端宽度接近全屏但仍保留边距。 视觉风格沿用博客当前的简约浅色系统：低饱和背景、轻边框、8-14px 圆角、少量阴影。

推荐交互流程

  1. 初始态

    右下角显示小机器人按钮。首次访问可用一个短 tooltip 提示“和我聊聊吧~”。

  2. 展开聊天窗

    顶部显示“和我聊聊吧~”和轻量说明；正文区展示 2-3 个建议问题；底部为输入框。

  3. 发送问题

    默认通过 POST /api/chat/stream 发送问题和可选 sessionId，SSE delta 只更新同一个回答气泡。

  4. 展示来源

    只有 RAG 命中博客内容时，答案下方出现来源卡片，点击跳转到文章 URL 或章节锚点。

  5. 错误与限流

    服务繁忙、限流、检索为空时使用短文案解释，不暴露内部错误或部署细节。

关键状态

空状态

展示能力边界和建议问题，例如“这个博客讲过哪些密码学内容？”“作者是谁？”

生成中

SSE 流式生成，但 UI 不拆成多条消息。输入框进入等待态，按钮显示“思考中”，避免重复提交。

RAG 无命中

不强行编造来源。提示“博客里没有找到可靠内容，我按通用知识回答”。

模型繁忙

私有模型服务资源有限，排队或返回“当前繁忙，请稍后再试”。不展示具体硬件信息。

3. 技术架构总览

现有博客继续保持静态部署。新增后端命名为 blog-service，只负责博客问答、RAG 检索、 限流和来源构造。模型计算能力不放进业务后端，而是沉到通用 AI 计算服务中；未来图片理解、代码助手、摘要生成、 批量 embedding 等 AI 需求都复用同一个内部 API。 

用户浏览器
  ├─ GET /                    → 静态博客页面
  └─ POST /api/chat/stream    → RAG API
                                ├─ 聊天限流器：同时只处理少量生成任务
                                ├─ 查询向量化 → AI 计算网关
                                ├─ Qdrant: blog_chunks collection
                                ├─ Rerank: vector + title + tag + updated_at
                                ├─ Prompt Builder: blog snippets + persona rules
                                ├─ 聊天生成 → AI 计算网关
                                └─ 引用构造：标题 + 小节 + 锚点 URL

blog-indexer 定时任务
  ├─ 每 1 分钟扫描 src/content/posts/*.md
  ├─ SQLite index status 做增量判断
  ├─ Embedding → AI 计算网关
  └─ Upsert/Delete → Qdrant

推荐服务命名

资源 推荐名称 说明
RAG API blog-service 面向博客问答的 Spring Boot 服务，不持有模型和硬件细节。
AI 计算网关 ai-compute-gateway CPU Pod，暴露 OpenAI 兼容 API，负责队列、鉴权、优先级和运行时适配。
模型运行时 ai-compute-runtime 私有模型运行时，底层跑 Ollama；具体硬件和资源申请只写在部署清单里。
向量库 qdrant 独立 StatefulSet 或 Deployment + PVC。
索引任务 blog-indexer k3s CronJob，每 1 分钟运行一次，增量更新 Qdrant。

4. 技术选型

层级 选型 理由 风险
后端 Spring Boot + Spring AI blog-service 和 ai-compute-gateway 都使用 Spring Boot，栈一致，便于维护。 Spring AI 版本迭代较快，落地前需锁版本并验证 API。
AI 计算网关 OpenAI 兼容 API 对内暴露 /v1/chat/completions、/v1/embeddings、/v1/models，业务服务不直接依赖 Ollama。 多一层服务，但后续替换运行时或增加队列更稳。
模型运行时 Ollama 第一版直接使用 Ollama 管理本地模型，业务层只感知统一网关。 私有算力有限，需要限制并发和输出长度。
向量库 Qdrant 独立服务、扩展性好、支持 payload 元数据和过滤，适合来源追踪。 只在集群内访问。暂不备份，可由 Markdown 重建。
索引 CronJob + CI 异步触发 blog-indexer 每 1 分钟增量扫描；CI 发布后异步触发一次 Job，但不阻塞博客发布。 页面可能先更新，索引稍后补齐，最坏由下一分钟 CronJob 兜底。
响应 JSON + SSE POST /api/chat 返回 JSON；默认前端使用 POST /api/chat/stream，同一个回答气泡内流式更新。 SSE 只影响显示速度，不能把 delta 渲染成多条机器人消息。
限流 SQLite 每 IP 10 次/分钟、100 次/天；日志不保存完整问题，只存 hash、长度、耗时和命中情况。 多副本时需共享 PVC 或改为 Redis；初期建议 blog-service 单副本。
重要约束： 你的现有博客是静态站，不能把模型 Key、RAG 检索或限流逻辑放在 Astro 前端。前端只能是聊天组件， 博客智能逻辑进入 blog-service，GPU 调度逻辑进入 ai-compute-gateway。

5. RAG 与意图识别设计

意图类型

类型 示例 处理链路 是否展示来源
persona 你是谁？这个站是谁写的？作者现在关注什么？ 读取人设资料 + 受控回答模板 否
blog_rag RSA 为什么需要欧拉函数？这篇文章讲了什么？ 查询向量化 → Qdrant 检索 → 上下文压缩 → 生成回答 是
general 解释一下 HTTPS；推荐一个学习路线。 直接调用本地模型，使用通用系统提示词 否
reject_or_clarify 请求隐私、攻击、无意义输入、过长输入。 拒答或追问 否

意图识别策略

初期不建议上复杂分类模型。更稳的方案是“规则优先 + 小模型分类兜底”：

  1. 规则命中人格问题

    包含“你是谁、作者是谁、这个站、yinsheng、联系方式”等关键词时进入 persona。

  2. 规则命中博客问题

    包含文章标题、标签、已知主题词，或用户明确说“博客里/文章中/这篇”时进入 blog_rag。

  3. 轻量 LLM 分类

    无法判断时调用本地模型输出 JSON：{ intent, confidence, reason }。

  4. 低置信度兜底

    低置信度时先做一次 RAG 检索；若得分低于阈值，按 general 回答且不展示来源。

切片建议

参数 建议 原因
切片粒度 按 Markdown 的 H2/H3/H4 分段后再按 500-900 中文字符切片 保留章节语义，来源卡片也能定位到标题。
重叠 80-150 中文字符 避免公式或推导链断裂。
召回数量 检索前 5 条，最多返回 5 条引用 符合博客问答场景，也不会让引用卡片淹没正文。
来源展示 /{slug}/#anchor 当前博客实际文章路由如此，点击 citation 在当前页跳转到对应小节。
增量规则： 索引器读取 src/content/posts/*.md，状态保存在 blog-indexer-data PVC 中的 SQLite。 新增文章写入新 chunk；修改文章先删旧 chunk 再写新 chunk；删除文章时根据 SQLite 中消失的 slug 删除 Qdrant 中所有对应点。

7. K3s 部署方案

新增组件不应破坏现有 blog-web。博客业务继续在 blog namespace； AI 基础设施放入独立 namespace。跨服务访问先默认允许，靠内部服务发现和 token 控制。

资源 类型 建议配置
blog-service Deployment + Service 业务 namespace，1 副本，暴露 /api/chat 和 /api/chat/stream。
blog-indexer CronJob + Job 业务 namespace，每 1 分钟执行，concurrencyPolicy: Forbid，挂载内容源和状态 PVC。
ai-compute-gateway Deployment + Service AI namespace，Spring Boot，OpenAI 兼容 API，负责队列和鉴权。
ai-compute-runtime Deployment + Service + PVC AI namespace，Ollama 私有模型运行时，挂载模型 PVC；具体资源申请只在 YAML 中维护。
qdrant StatefulSet + Service + PVC AI namespace，1 副本，持久化 blog_chunks，仅集群内访问，不启用 API key。
blog-content-source PVC 业务命名空间，由 GitHub Actions runner 写入 Markdown 源文件，索引器只读挂载。
blog-indexer-data PVC 业务 namespace，保存 SQLite 状态表 blog_index_status。
blog-service-data PVC 业务 namespace，保存聊天限流 SQLite，仅记录 IP hash 和时间窗口。

模型部署建议

生成模型

主模型 qwen2.5:3b。qwen3:4b 需确认不会流式输出思考过程后再启用。Chat 最大输出 1000 tokens，同一时间只允许 1 个生成任务。

Embedding 模型

主模型 bge-m3，备用模型 nomic-embed-text。Embedding 也走 Ollama， 但必须服从 chat > embedding 优先级。

注意： 只有私有模型运行时可以使用本地加速资源。RAG API、索引器、Qdrant 和网关都不能申请这类资源。

8. CI/CD 与索引流程

现有 CI 已优化为依赖未变化时跳过安装，并使用 npm exec -- astro build 构建。 新增聊天入口后，CI 只负责发布静态文件、同步 Markdown 源文件，并异步触发索引 Job，不等待索引完成。

  1. 构建 Astro

    仅当 src/**、public/**、依赖或构建配置变化时触发部署 workflow。

  2. 发布静态文件

    将 dist/ 增量同步到 blog-public PVC，nginx 立即读取新页面。

  3. 同步 Markdown 源文件

    将 src/content/posts 同步到 blog-content-source PVC，供 CronJob 读取。

  4. 异步触发索引

    CI 创建一次 blog-indexer Job 后直接结束；索引失败不影响博客发布。

  5. CronJob 兜底

    每 1 分钟执行一次增量扫描，确保遗漏或失败能自动补上。

CI 后置动作示例

# 同步 Markdown 到 PVC 挂载点
rsync -a --delete src/content/posts/ /deploy/content/posts/

# 异步触发一次索引器 Job，不等待完成
kubectl -n blog create job --from=cronjob/blog-indexer blog-indexer-manual-${GITHUB_RUN_ID} || true

9. 安全、隐私与成本控制

限流

SQLite 表记录 ip_hash、分钟窗口、日窗口、失败次数。初期建议 10 次/分钟、100 次/天。

日志

只记录请求量、耗时、意图、是否 RAG 命中、错误码。不保存完整问题正文。

防滥用

限制输入长度、RAG 上下文最大 5000 tokens、输出最大 1000 tokens。

来源约束

只有 RAG 命中且分数过阈值时展示来源，避免伪来源。

人设保护

人设资料放在 config/persona.md，以 ConfigMap 挂载，不暴露原文；遇到隐私请求应拒答。

公网暴露

公网只暴露博客和 /api/chat*；Qdrant、网关、运行时、索引器均不直接暴露公网。

内部鉴权

RAG API 调用网关需要内部服务 token；向量库暂不启用 API key，仅集群内访问。

队列策略

网关中聊天生成的优先级高于 embedding；排队超过 30 秒返回 429。
